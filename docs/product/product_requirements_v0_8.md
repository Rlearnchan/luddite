# Product Requirements v0.8

## 1. 우선순위

사용자 응답 기준 우선순위:

```text
1. E. 리서치팀의 주제 선별 보조
2. A. 월~금 아침 후보 digest
3. B. 수집된 뉴스 DB 기반 storyline 초안
4. C. PPT 초안은 장기 접근
```

## 2. 실제 업무 흐름

리서치팀 주간 흐름:

```text
월/화: 자료 수집 및 스토리 작성
수/목/금: PPT 제작
```

근무 흐름:

```text
형찬: 9 to 6
동찬/성원: 오후 출근 ~ 늦은 밤
```

따라서 Luddite는 특정 시간 한 번만 발행하는 newsletter보다, 다음 두 형태가 모두 필요하다.

```text
1. 월~금 오전 digest
2. 조회 시점에 Slack/Sheet에서 후보 검색/재생성
```

## 3. 첫 데모 정의

성공했다고 느끼는 첫 데모:

```text
매일 아침 후보 digest가 나온다.
```

기대되는 digest:

```text
- 후보 10개
- 클릭하고 싶은 제목
- seed 기사 링크
- 왜 슈카월드감인지
- 가능한 3단 확장
- 추가로 찾을 자료
- 리스크/보류 이유
- 담당자가 시트에서 상태를 바꿀 수 있음
```

## 4. 초기 UI

선호 UI:

```text
1. Google Sheet에 결과 쓰기
2. Slack/메일 알림 또는 조회
```

초기 구현 추천:

```text
- Google Sheet: 후보 저장/상태 관리
- Slack: 오늘의 후보 10개 조회, 키워드 검색, 후보 상세 보기
- Markdown report: 내부 기록 및 debugging
```

웹앱은 후순위.

## 5. 비용/속도

운영 기준:

```text
- 하루 1회 batch면 충분
- 시간은 오래 걸려도 됨
- API 비용은 일 $1 미만이면 이상적
```

따라서 실시간보다는 batch + cache + incremental 방식이 적합하다.

## 6. 품질 기준

### jibi

좋은 jibi 결과:

```text
- 사람이 클릭해보고 싶다.
- 왜 미사용/보류인지 설명이 납득된다.
- 좋은데 위험한 소재를 무리하게 A로 밀지 않는다.
- 해외 seed를 국내 연결 없이도 구조적으로 평가한다.
```

### anny

좋은 anny 결과:

```text
- 3~4단 구조가 자연스럽다.
- headline이 바로 PPT로 옮길 수 있다.
- 출처가 빠지지 않는다.
- 장황한 리포트가 아니라 방송용 outline이다.
```

한국 연결은 중요하지만 필수는 아니다. 마지막 관심 환기 정도면 충분한 경우가 많다.

### piti

좋은 piti 결과:

```text
- speaker notes에 정확한 출처가 보존된다.
- 슬라이드당 한 메시지다.
- 슈카월드 포맷에 가깝다.
- 사람이 열어서 바로 고칠 수 있다.
```
